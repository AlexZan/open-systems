A post is just content + a parent. Therefore any content can be the
parent or child of any other content. A post can be made on a
suggestion.

A post can have multiple parents and multiple children; a networking of
linked content. When a post is made, a user can specify what parents it
has.

If a post can have any parents. What if the parent is a bounty, then the
post will be a reply to the bounty and thus a potential solution. This
opens the doors to a post being reused as solutions to multiple
bounties. In this case the votes are not made on the post content
itself, but the post content in relation to the bounty; the parent. This
now ads complexity since there are now multiple sets of votes, one for
the post iself, the other for the post in relation to another post.

Maybe then all votes must be inherently contextual, and so a post must
always be voted in relation to another post. Something that needs to be
explored.

Use cases:

A utility library is used as a part solution for multiple
bounties/goals. But the solution is a post describing why the library
would be a good fit, not the actual library post itself. Each time its
posted as a solution, it is a custom post.

In this case, and any others i can think of, it makes the reuse of post
as solutions impractical, since they will usually be new posts
describing how it fits the context. This might be overdesign.

In this case then maybe a post should only be allowed one parent, but it
can reference as many

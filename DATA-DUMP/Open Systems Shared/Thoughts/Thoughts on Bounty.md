## Distribution

If a single award is set when the bounty is awarded, then distribution
is not required, it will all go just to one user.

If a distributed award is selected, then the tags that the funds to must
be selected. Maybe they could be the same tags which the bounty gets
voted on if its completed. Yes, the distribution can be based off the
tags voted for the bounty.

If users get to split the award depending on their vote count, then
users might not be as worried about being the main winner since they
will get something regardless. The main winner will get less, but they
will still get the most so it is good motivation regardless. Also by
distributing awards based on votes, it encourages more cooperation since
users will race to be the first to respond with partial solutions, and
other users can continue to build on their solution. If it was not
distributed, then everyone would keep their effort and value to
themselves until the last minute. It would cause a lot of duplicate
efforts without knowing. **Therefore the reward should always be
distributed.** Would then a single reward ever be needed?

## Consensus on Completion and Triggering Reward

#### Use Cases Bounty

Open Forum feature request (e.g. a built in wallet in the browser to
make funding easier) as an open post, a bounty is replied to the post.
Users begin to fund the bounty. A user votes a tag; #blockchain. Another
votes another tag #blockchain wallet, then #programming, #open forum
development, #open forum api as other tags get voted in.

A programmer creates the request feature and replies to the bounty
linking the git pull request. The tags from the bounty are automatically
inherited. Users begin to upvote the appropriate tags they have
experience in. Another that was also working on their own version posts
their git pull request a day later, explains how they took a different
approach. An experienced programmer response to the new proposal,
explains why they think its a better solution. Other programmer users
upvote the response and the solution, indicating the point is valid, and
that it is the better solution. A few days later after several debates,
the second submission has clearly more votes. The programmer from the
first even replies to the second and agrees that it is a better
solution.

Now that a clear winner has emerged, what should trigger the bounty
funds to be sent? How do we go from just voting a tag on a solution, to
collectively deciding it fulfills the bounty?

It must be directly trigger by vote, a vote that indicates the user
thinks the bounty is complete, or that a solution should be awarded. An
idea is to be able to vote which solution each user thinks is the
winner, and the vote count for each user that posted the solution will
be the weight to the bounty award. This causes the possibility that the
solution with less experience votes, gets a higher award, which i dont
think makes sense. This also does not work with bounty reward
distribution based on votes.

The other option is to vote on the bounty itself that it has been
completed. In which case a threshold must be established in order to
trigger the completion and reward. What is the threshold? Tag votes have
no upper limit established to reach a threshold.

Maybe every bounty tag has a "vote completed" button. When the votes
reach half of the total votes on that tag, that tag is completed. When
all the tags are voted completed, then the bounty is completed.

Maybe by voting on a bounty, you are investing to get a return, if the
bounty passes?

#### Use Case

Open forum feature Bounty is made. #open forum api, #programming, and
some other tags are voted. Another user thinks this is a good feature
and so they upvote all the tags they have experience in. with several
votes in each, since they are so confident of a return. Several
responses are made with content leading towards a solution. A solution
is posted and it and the other responses get several upvotes in their
pertaining tags. A few days pass and users start to vote on the bounty
that the tags are completed. All tags are voted as completed and the
bounty is then completed.

#### Use Case : Wrong tag

Bounty is made about an open forum feature. Someone votes the tag
#comedy with 10 votes. This person has invested 10 votes hoping he can
make a return. But because its a vote that has no relevance, people will
downvote comedy because they know its wrong and they can get a return on
their downvotes. 20 votes are made against #comedy downvoting it. It now
has -10 votes and can no longer be downvoted. It is now removed from
showing up by default in the bounty. 24 hours reaching the downvote
limit, the users who downvoted it double up their experience. The user
who upvoted it loses all 10 experience.

#### Use Case: 1 vote tag

Bounty is made about an open forum feature. Someone votes #forum as a
tag with 1 experience. It is not a very relevant tag, but it is not
incorrect either. It remains at 1 exp during the entire time of the
bounty. After a solution is replied, the other tags are voted as
completed.

Someone must vote 1 tag of #forum, but it\'s obscure and no one else
might have experience in that tag, it might hold up the process of

So maybe a tag vote must get more than vote tag count / 100 \* greatest

than 100 votes, it does not need to get voted completed in order to be

##### What incentive is there to vote for a bounty tag?

Just like a content tag vote; you double the experience you voted when
there are more than double votes the vote count of when you voted.

What incentive is there to vote for a tag completed. The incentive is
that the bounty is completed eventually, and the users who contributed
value are rewarded, this allows the system to continue to keep
functioning, and for new features to keep being developed. The incentive
is long term, that the system will keep improving with your vote.
